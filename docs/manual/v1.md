# Tinkabot v1 User Manual

Tinkabot runs generated work — scripts and generated browser content — through a trusted substrate instead of handing it raw authority. You define scripts as durable records, wire them to triggers, and read the results as durable materials. NATS remains the substrate seam: every trigger, status, projection, and artifact is visible through ordinary NATS surfaces. The release package now also includes `tinkalet`, the profile-aware product CLI for local profile import and bundle trigger tours; the `nats` CLI remains the low-level operator and diagnostic tool.

Usage flow: https://diashort.apps.quickable.co/d/7c85909d

## Status

Every command and JSON shape in this manual is quoted from executed v1 proofs (`release/v1.json` is the evidence map; `bun run release:evidence` verifies it). The repo now contains the single-binary server surface at `substrate/go/cmd/tinkabot` and the profile-aware client surface at `substrate/go/cmd/tinkalet`; local package builds can use `bun run pack:tinkabot`, and release-shaped proof uses `bun run gate:tinkalet-package`.

## Concepts

**Principals and leases.** Every actor — caller, router, script runtime, observer, browser session — is a named principal holding a scoped lease, never a durable ambient credential. A lease has an id and a status (`active`, `revoked`, `expired`); the lease id is the credential. Deny wins over allow, and adjacent authority is rejected before effect.

**Scripts.** A script is a versioned JSON record in a KV bucket: a process contract (command, args, timeout, resources, cleanup), not NATS-aware code. Scripts talk to the platform only through framed stdio effects; the platform decides what materializes.

**Activation.** Triggers — request/reply, subject messages, KV changes, Object Store changes, stream messages, schedule ticks — are normalized into durable activation records with dedupe, cursors, lease binding, and loop suppression before any script runs.

**Materials.** Accepted script effects become durable truth: projections and artifact manifests in KV, artifact bodies in Object Store, attributed event envelopes in the ledger. Script output is never product truth until the materializer accepts it.

## Operating the platform

The substrate embeds a NATS server with JetStream. From Go:

```go
rt, err := embednats.Start(ctx, embednats.Config{
    ServerName:   "tb-main",
    Host:         "127.0.0.1",
    Port:         -1,                  // auto-assign
    StoreDir:     "/var/lib/tinkabot", // JetStream persistence
    ReadyTimeout: 2 * time.Second,
    StopTimeout:  2 * time.Second,
    Auth:         operatorAuth,        // core.Auth
    AuthUsers:    principals,          // every caller/router/runtime/observer lease
    WebSocket:    embednats.WebSocket{Enabled: false},
})
```

Each `core.Auth` user needs a non-empty `User`, an `active` lease (`Capability.LeaseID`, `Capability.LeaseStatus`), and NATS-shaped permissions (`Permissions.Publish.Allow/Deny`, `Permissions.Subscribe.Allow/Deny`). Request/reply principals additionally set `AllowResponses.Max` with a mandatory `AllowResponses.ExpiresMs`. `rt.Posture()` reports readiness, the client URL, JetStream state, single-node topology, and WebSocket posture. `rt.Connect()` gives in-process components a router-safe connection.

## Starting the binary

The single `tinkabot` binary assembles the proven pieces — embedded NATS with JetStream in operator/JWT mode, the embedded frontend shell, and the script materializer loop — behind a declared loopback exposure posture:

```bash
tinkabot --store /var/lib/tinkabot --shell 127.0.0.1:8419
```

For a local release-shaped package:

```bash
bun run pack:tinkabot
./dist/tinkabot/tinkabot --store /var/lib/tinkabot --shell 127.0.0.1:8419
```

The package layout includes a bundled Bubblewrap sidecar at `libexec/tinkabot/bwrap`. Runtime lookup order is `TB_BWRAP`, then the bundled sidecar, then `PATH`; the sandbox still preflights and fails closed if the host cannot run Bubblewrap namespaces.

First start materializes the substrate state into the store directory: the master operator key (`operator.nk`) and a short-lived creds file per manual role — `caller.creds`, `observer.creds`, `author.creds` — minted under the operator's app account. A restart reloads the same operator identity from `operator.nk` without regeneration and mints fresh role creds; corrupt key material fails typed instead of silently minting a new authority.

On start the binary prints its served posture: the loopback NATS client URL, the shell URL, and the role creds file paths. Every command in this manual runs against it in creds mode (`--creds "$STORE/caller.creds"`, see [Connecting with the nats CLI](#connecting-with-the-nats-cli)). The wiring posture names the served surfaces: the trigger and events subjects and the config, upload, script, ledger, material, and artifact stores.

The embedded shell is served on the loopback shell address under the narrow service-worker session scope `/__tinkabot_session/` with the proven policy headers (`Service-Worker-Allowed` bound to that exact scope, `Cache-Control: no-store`, `X-Tinkabot-Worker-Rev`). Stopping the binary (SIGINT/SIGTERM) drains: the trigger route and materializer loop stop, the role creds minted by the process are revoked — live credentialed connections are kicked and denied on reconnect — and the shell stops serving before the embedded server exits.

## Operating with Tinkalet

The release package includes `tinkalet` beside `tinkabot`. When `tinkabot`
starts, it writes `<store>/local-profile.json`, a local-owner descriptor with
the loopback NATS URL, shell URL, and caller credential reference. Tinkalet
imports that descriptor, copies the caller credential into its managed data
directory, and lets the user trigger a product intent without raw NATS subject
names:

```bash
export STORE=/tmp/tb-clock
export TINKALET_CONFIG_DIR=/tmp/tinkalet-config
export TINKALET_DATA_DIR=/tmp/tinkalet-data

./tinkabot --store "$STORE" --shell 127.0.0.1:8419 --bundle examples/clock
```

In another terminal, from the package root:

```bash
./tinkalet profile import local --store "$STORE" --name local
./tinkalet profile use local
./tinkalet trigger bundle.clock.tick --request-id req-clock-1
```

The trigger prints `profile local accepted bundle.clock.tick`, and the clock
projection served by the shell changes. `bun run gate:tinkalet-package` proves
this from the release archive with explicit Tinkalet config/data dirs and a
scrubbed Tinkalet environment; it renames the packaged NATS CLI sidecar before
running the Tinkalet commands, so the proof cannot be satisfied by shelling out
to `nats`.

## Connecting with the nats CLI

The platform mints each principal a short-lived credentials file — a user JWT signed under the substrate's master operator key, carrying the principal's lease. Connect with it:

```bash
nats --no-context --server "$CLIENT_URL" --creds caller.creds --timeout 2s <command>
```

Revoking a principal disconnects its live connections and denies reconnect; permission updates roll onto live connections without a restart. Permission denials still happen before any effect. When the substrate is embedded without operator mode, the static form remains: `--user <principal> --password <lease-id>`.

## Defining a script

A script record lives in the script KV bucket under key `s.<base64url(scriptKey)>` (e.g. `scripts.proof.echo` → `s.c2NyaXB0cy5wcm9vZi5lY2hv`). The record is strictly decoded — unknown fields are rejected:

```json
{
  "kind": "script.record",
  "scriptKey": "scripts.proof.echo",
  "scriptRevision": 7,
  "desc": "Render the main material projection.",
  "process": {
    "command": "/bin/sh",
    "args": ["-c", "emit-effects"],
    "cwd": "/tmp/work",
    "rpc": "framed_stdio",
    "timeoutMs": 2000,
    "resource": { "cpuMillis": 100, "memoryMB": 64 },
    "kill": "process.kill",
    "cleanup": "workdir.delete",
    "identity": "principal.script.001"
  }
}
```

The script process emits effects as length-framed JSON on stdout (`Content-Length: <n>\r\n\r\n<body>`), one frame per effect. Three effect types exist, each strictly validated:

```json
{ "kind": "script.effect", "effectType": "projection",
  "projectionId": "main", "snapshotRevision": "snap-001",
  "artifactRevision": "artifact.rev.7", "sequence": 1,
  "value": { "title": "from-script" } }
```

```json
{ "kind": "script.effect", "effectType": "artifact",
  "artifactName": "artifact/main.js", "artifactRevision": "artifact.rev.7",
  "mediaType": "application/javascript", "body": "export default 1" }
```

```json
{ "kind": "script.effect", "effectType": "publish",
  "subject": "tb.app.events.example", "body": "payload" }
```

The script never sees NATS credentials, subjects beyond its grants, or store handles. Malformed frames fail the run with attribution; nothing materializes.

## Triggering work

All triggers pass source authority, then the durable ledger, before a script runs. The reply or recorded outcome is always a status: `accepted`, `duplicate`, `LeaseRevoked`, `StaleCursor`, `DeniedNeighbor`, `LoopSuppressed`, or `SourceMalformed`.

**Request/reply** — identity travels in a header; the reply is the ledger status:

```bash
nats request --raw -H Tinkabot-Request-Id:req-rel-001 tb.proof.runtime.execute ping
# -> accepted
```

Repeating the same `Tinkabot-Request-Id` returns `duplicate` and runs nothing.

**Subject message**:

```bash
nats publish -H Tinkabot-Message-Id:msg-rel-001 tb.proof.runtime.execute ""
```

**KV change** — a watched bucket/key activates on write; the KV revision becomes the source position:

```bash
nats kv put config_bucket app_config '{"state":"new"}'
```

**Object Store change** — a watched bucket/name activates on upload, preserving the meta-stream sequence:

```bash
nats object put artifacts ./app.bundle.js --name app.bundle.js
```

**Stream message** — a stream/consumer pair activates per message with stream and consumer sequences:

```bash
nats publish tb.proof.events.main event
```

**Schedule** — engine-driven, not message-driven. The activation schedule engine accepts deterministic clock ticks under a leader lease with fencing; due ticks become activations, duplicates and rewound clocks are denied, and restarts catch up only missed ticks from durable JetStream KV state. Tinkalet product schedules are separate server-owned interval records that write `tinkabot.item.v1` ticks for humans, scripts, and local daemons.

## Observing results

Materials land in the material KV bucket and artifact Object Store, readable with observer credentials:

```bash
# Projection (key p.<projectionId>)
nats kv get "$MATERIAL_BUCKET" p.main --raw
# -> { "kind": "material.projection", "projectionId": "main", "sequence": 9,
#      "snapshotRevision": "...", "artifactRevision": "artifact.rev.7",
#      "value": { "title": "from-script" }, "observedAt": "...", "provenance": { ... } }

# Artifact manifest (key a.<base64url(name)>; artifact/main.js shown)
nats kv get "$MATERIAL_BUCKET" a.YXJ0aWZhY3QvbWFpbi5qcw --raw
# -> { "kind": "artifact.manifest", "digest": "sha256:...", "mediaType": "...", ... }

# Artifact body
nats object get "$ARTIFACT_BUCKET" artifact/main.js --output ./main.js

# Event envelopes live in the ledger bucket under e.<eventId> with
# status success|failed, principal, capability, chain, and typed error.
```

Every result carries provenance (`schemaId: tb.schema.base.contract_authority.v1`, app revision, producer) and chain attribution, so any outcome traces to one owning layer.

## A worked example: ship a status page

Everything above composes into one loop. The binary boots with a single wired script slot: the trigger subject runs `scripts.app.main` at revision 1, and the runtime grants that script exactly one projection (`main`) and artifact paths under `artifact/` — anything else a script emits is denied with attribution. This example replaces the wired script with a status-page renderer and ships the page, end to end, using only the three role credentials the binary minted at startup.

The script is `docs/manual/examples/status-page.json`: a plain `/bin/sh` process that emits two framed effects — the `main` projection carrying status state, and an HTML artifact at `artifact/status.html`. In a real deployment this would be your health checker, report builder, or page generator; the platform contract is identical.

The author lands the record (the author role can write only the script bucket; the put echoes the stored record):

```bash
nats kv put "$SCRIPT_BUCKET" s.c2NyaXB0cy5hcHAubWFpbg "$(cat docs/manual/examples/status-page.json)"
# -> "scriptKey": "scripts.app.main", ..., "scriptRevision": 1, ...
```

The caller renders the page. Acceptance is durable, so re-sending the same request id is safe and runs nothing:

```bash
nats request --raw -H Tinkabot-Request-Id:req-status-001 tb.proof.runtime.execute render
# -> accepted

nats request --raw -H Tinkabot-Request-Id:req-status-001 tb.proof.runtime.execute render
# -> duplicate
```

The observer reads the shipped state. The projection is the page's data, the manifest is its content-addressed identity, and the body is the page itself:

```bash
nats kv get "$MATERIAL_BUCKET" p.main --raw
# -> { "kind": "material.projection", ..., "sequence": 10,
#      "value": { "page": "status", "state": "ok", "checks": 3 }, ...,
#      "producer": "script-materializer" }

nats kv get "$MATERIAL_BUCKET" a.YXJ0aWZhY3Qvc3RhdHVzLmh0bWw --raw
# -> { "kind": "artifact.manifest", "artifactRevision": "page.rev.1",
#      "digest": "sha256:1eac1cbe...", "mediaType": "text/html", ... }

nats object get "$ARTIFACT_BUCKET" artifact/status.html --output /tmp/tb-status.html --force >/dev/null 2>&1 && cat /tmp/tb-status.html
# -> <h1>status: ok</h1>
```

What just happened, in platform terms: the author's write was permitted only on the script bucket; the caller's trigger passed source authority and the durable ledger before the script ran; the script never saw NATS — it printed framed JSON to stdout and the runtime's grants decided what could materialize (a `projectionId` other than `main` fails as `FacadeEffectDenied`, exactly as a wrong `scriptRevision` fails as `ScriptRevisionMismatch`, each recorded as a failed `event.envelope` with the error attributed to its owning layer); and the observer could read everything but write nothing. Replace the one-liner with a real program and this is the production shape.

## Authority boundaries you will hit

These denials are proven behavior, not advice:

- A **caller** cannot write the ledger KV bucket — `nats kv put <ledger> ...` is denied by permissions.
- An **observer** cannot write material KV or publish Object Store chunks (`$O.<bucket>.C.*`) — read-only by construction.
- A **neighbor subject** one token away from your grant is denied before effect; `deny` entries beat `allow` entries.
- A **revoked lease** stops working immediately: live request/reply with a revoked source lease returns `LeaseRevoked`.
- Note: `nats` CLI v0.3.0 reports permission errors in command *output* while exiting 0 — check output text, not exit codes.

## Browser path

Generated browser content never touches NATS. The trusted shell runs generated artifacts in an opaque sandboxed iframe (`allow-scripts` only, never `allow-same-origin`), binds messages by source window, nonce, frame lease, schema/artifact revision, and a command allowlist, and rejects raw-authority vocabulary outright.

Accepted intents go to the gateway as durable commands on `tb.app.browser.command`:

```json
{ "commandId": "cmd-nats-001",
  "gateway": { "sessionId": "session-001", "leaseId": "lease-001",
               "csrfToken": "csrf-001", "origin": "https://app.localhost",
               "fetchSite": "same-origin", "fetchMode": "cors", "fetchDest": "empty",
               "artifactRevision": "artifact.rev.7", "credentialedCors": false } }
```

The reply is a schema-validated `command.acceptance` status (`accepted` or `rejected` with a typed, attributed error). The gateway denies bad CSRF, wrong origin, bad fetch metadata, credentialed CORS from generated origins, stale artifact revisions, and revoked leases. Service-worker setup is server-owned and cookie-session-backed: the worker script is served under an exact allowed scope with `Service-Worker-Allowed`, `Cache-Control: no-store`, and a worker revision header; generated content gets no tokens and cannot register workers.

## Agent wrapper (local proof only)

The `substrate/go/apps/wrapper` package drives the real `claude` CLI over
structured stream-json stdio, connects as a trusted-wrapper principal minted
by `MintTrustedWrapper` (publish allow on `tb.session.<id>.ingest`, subscribe
allow on `tb.session.<id>.steer`, nothing else), republishes the agent's
output as canonical `session.frame` envelopes through the frame-mediation
path, and translates canonical `session.steer_intent` payloads from the steer
subject into claude stream-json user messages on the agent's stdin.

Frame shape on the output stream `tb-session-out-<id>`: streaming text deltas
arrive as token frames (`{"kind":"session.frame","frame":"token","origin":
"wrapper","sessionId":...,"text":...}`); every other agent event arrives as a
chunk frame whose `body` is the verbatim event line as a string value (agent
event keys such as `usage.input_tokens` collide with the reserved-vocab
facade, which scans property names, never values).

**Manual-runner coupling, named per Plan slice 6:** `gate:manual` executes
only `# ->` pairs against the running `tinkabot` binary, and the session
subsystem is not part of the binary surface until Slice 7 registers it. The
pairs below therefore use `# Expected:` annotations (not executed by the
gate), and their validation is owned by the scripted equivalents: live runs
of `TestAgentWrapperMediated` (CI-runnable, real embedded NATS, recorded
frames) and `TestAgentWrapperLocalE2E` (local-only live agent, by the
no-live-agent-in-CI carried decision).

### Wrapper lane pair (validated 2026-06-12)

```bash
cd substrate/go && go test ./apps/... -count=1
# Expected: ok — decode over recorded real-CLI stream-json lines (typed
# StreamJsonParseFailure on malformed input), canonical token/chunk frame
# build validated against the slice-1 contract registry, steer translation.

cd substrate/go && go test ./embednats -run 'TestAgentWrapperMediated' -count=1
# Expected: PASS — the real wrapper loop under a MintTrustedWrapper credential
# over real embedded NATS in operator/JWT mode: canonical frames observed on
# the mediated output stream, a malformed line survived, a canonical steer
# intent delivered to the subprocess stdin in claude user-message shape.
```

### Live agent pair (local only)

Prerequisite: `claude` CLI authenticated locally. The required flag set is
`--print --verbose --input-format stream-json --output-format stream-json
--include-partial-messages` (`--verbose` is mandatory: the CLI rejects
`--print` with stream-json output without it).

```bash
cd substrate/go && TB_E2E_CLAUDE=1 go test ./embednats -run 'TestAgentWrapperLocalE2E' -count=1 -v
# Expected: PASS — a live claude session is opened by a first steer intent
# ("Reply with exactly the word: pong"), its token frames are observed on the
# mediated output stream, then a second steer mid-session ("Now reply with
# exactly the word: maple") produces the steered tokens — proving the session
# is observable and steerable while alive.
```

### Standalone wrapper operation

With a running substrate (operator mode), a trusted-wrapper creds file, and
an observer credential:

```bash
go run ./apps/wrapper/cmd/run \
  --nats "$CLIENT_URL" \
  --creds "$WRAPPER_CREDS" \
  --session "$SESSION_ID" \
  -- claude --print --verbose \
            --input-format stream-json \
            --output-format stream-json \
            --include-partial-messages

# Steer with the canonical intent (the wrapper skips any other payload kind):
nats --no-context --server "$CLIENT_URL" --creds caller.creds \
     pub "tb.session.${SESSION_ID}.steer" \
     "{\"kind\":\"session.steer_intent\",\"intent\":\"steer\",\"sessionId\":\"${SESSION_ID}\",\"text\":\"Reply with exactly the word: pong\"}"
# Expected: token frames carrying "pong" on the output stream.

nats --no-context --server "$CLIENT_URL" --creds observer.creds \
     stream view "tb-session-out-${SESSION_ID}"
# Expected: canonical {"kind":"session.frame","frame":"token"|"chunk","origin":"wrapper",...}
# envelopes; never a wrapper-origin status frame (the FrameMediator rejects those).
```

### Denial pair

The wrapper credential cannot publish on the steering subject or on a
neighboring session's ingest subject. The Go owners of these denials are the
Slice 4 `TestTrustedWrapperAuthority` sub-tests (SelfDeclaredTrust and the
denied-neighbor proof); the CLI form:

```bash
nats --no-context --server "$CLIENT_URL" --creds "$WRAPPER_CREDS" \
     pub "tb.session.${SESSION_ID}.steer" '{"kind":"session.steer_intent","intent":"steer","sessionId":"x","text":"hijack"}'
# Expected: Permissions Violation for Publish to "tb.session.<id>.steer"
# (the nats CLI reports permission errors in output, not exit code)

nats --no-context --server "$CLIENT_URL" --creds "$WRAPPER_CREDS" \
     pub "tb.session.OTHER-SESSION.ingest" '{"kind":"session.frame","frame":"token","origin":"wrapper","sessionId":"OTHER-SESSION","text":"x"}'
# Expected: Permissions Violation for Publish to "tb.session.OTHER-SESSION.ingest"
```

## v1 limits

Named in `release/v1.json` as deferred, not proven: direct browser NATS WebSocket, Docker sandboxing of script processes (v1 runs trusted local processes), product UI rendering (the shell is a proof surface), live auth reload, activation-engine wall-clock loops beyond the Tinkalet item scheduler, broad script CRUD UI, multi-node HA (v1 is single-node; HA exists at contract shape only), and package publication.
